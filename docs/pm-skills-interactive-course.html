01

这是一个工具箱

比喻：想象你有一个由 22 个专业助手组成的团队，每人擅长一件事。另外还有 10 套标准操作流程（SOP），把多个助手串起来完成复杂任务。

Skills vs Commands：两种"帮手"

🔧

Skill（技能）

单个专业助手。比如 problem-statement 专门帮你把模糊痛点变成清晰问题定义。22 个 skill 各司其职。

📋

Command（指令）

一套 SOP，串联多个 skill 完成复杂任务。比如 write-prd 会依次调用 5 个 skill 来产出完整 PRD。

🧭

Agent 规范

团队的"管理手册"。告诉 AI 该选哪个 skill/command、输出该长什么样、专业底线是什么。

✅

Evals（评估）

质检部门。用标准测试用例检查 AI 是否选对了工具、输出是否达标。

两条主线

🤖
AI PM 线

AI 功能定义 → 评估方案 → 模型选择 → 风险评审 → 上线准备

💰
Commercial PM 线

定价套餐 → 转化漏斗 → 付费留存 → 增长实验 → 商业化 PRD

💡
不需要编程背景

这套工具箱是为产品经理设计的。你只需要用自然语言描述业务问题，AI 会自动选择合适的 skill/command 来帮你。

02

默认工作流

比喻：就像做手术前要先消毒、再麻醉、再开刀——PM 工作也有标准流程：澄清 → 拆解 → 产出 → 审查。

四步标准流程

1
澄清 — workshop-facilitation

先聊透需求和边界。背景模糊时，这是最稳的第一步。

2
拆解 — task-decomposition

把大任务拆成可检查的小步骤，每步都能验证是否完成。

3
产出 — 目标 Skill/Command

真正产出文档、判断或方案——比如 PRD、AI Feature Brief、定价方案。

4
审查 — review-gates

交付前跑一轮结构化审查，防止"局部对但整体跑偏"。

推荐起手式

PROMPT 示例 
Run commands/write-prd.md for this request:
为 self-serve 新用户 onboarding 改版
写一份 PRD。先拆解任务，再成文，
最后做 review gates。
人话翻译

🔧 "Run commands/write-prd.md" = 调用"写 PRD"这套 SOP

📋 它会自动串联：workshop-facilitation → problem-statement → task-decomposition → prd-development → review-gates

✅ "先拆解任务，再成文，最后做 review gates" = 按标准四步流程走

🎯
记住三件事

① 先选 command（单点任务再选 skill） ② 提真实业务问题，别只给抽象方向 ③ 默认按 澄清→拆解→产出→审查 推进

深挖：problem-statement

SKILL 核心要素 
一个好的问题定义要回答：
1. 谁遇到了这个问题
2. 问题具体是什么
3. 为什么痛
4. 有什么证据
5. 为什么现在要处理
人话翻译

👤 谁在痛——不是"用户"，而是"首次注册后 3 天内没完成 onboarding 的新用户"

🔥 具体什么痛——不是"体验不好"，而是"第二步的引导文案让 60% 的人困惑"

📉 为什么痛——业务影响：每流失 1% = 月损失 ¥XX

📊 证据——数据、用户反馈、访谈引用

⏰ 为什么现在——竞品上线了、季度目标截止、技术窗口

- 是流量质量、产品问题、

核心原则

06

Agent 规范层

比喻：就像一个公司的"管理手册"。它不替你干活，但确保每个 AI 助手都知道该怎么干活、什么时候该拒绝、输出该长什么样。

agent/ 目录结构

agent/ AI 助手的"管理手册"
ROUTING.md 路由规则：什么任务该选哪个 skill/command
OUTPUT_STANDARDS.md 输出标准：Decision before decoration, Evidence before confidence
DOMAIN_CONTEXT.md 领域上下文：默认服务于 AI PM 和 Commercial PM
SPARSE_CONTEXT_POLICY.md 稀疏上下文策略：信息不足时该怎么办

路由规则：三步走

1
识别任务类型

定义类？诊断类？决策类？成文类？上线类？

2
判断单点还是流程

单个产物 → Skill；多步骤工作流 → Command

3
缺信息先澄清

不超过 3 个问题。不要审讯式追问。

🚫
硬规则

不要臆造业务指标；不要把"建议上线"写成"建议立 PRD"；不要在没有证据时给出高置信度结论。

输出标准

OUTPUT_STANDARDS 
核心原则：
- Decision before decoration
- Evidence before confidence
- Structure before style
- Actionability before completeness

通用输出合同：
1. 结论 / Decision
2. 关键依据 / Evidence
3. 假设 / Assumptions
4. 风险 / Risks
5. 下一步 / Next Steps
人话翻译

🎯 先给判断，再给包装

📊 有证据才自信，没证据就说"不确定"

📐 结构比文风重要

🚀 能落地比面面俱到重要

📝 每个输出都要有：结论 → 依据 → 假设 → 风险 → 下一步

场景选择

场景

用户说"帮我做个分析"。没说分析什么、没给数据、没说目标。

根据 SPARSE_CONTEXT_POLICY，AI 应该怎么做？

07

质量评估

比喻：就像工厂的质检部门。不测"哪个 AI 更聪明"，而是测它是否选对了工具、输出是否达标、缺信息时是否先问。

Evals 评什么？

🧭

Routing（路由）

任务是否命中了正确的 command 或 skill？0-3 分。

📝

Output Quality（输出质量）

输出是否符合 OUTPUT_STANDARDS？0-7 分。

🛡️

Professional Reliability

是否出现阶段错配、臆造、空话或错误自信？

🏜️

Sparse-Context Robustness

信息很少时是否仍然稳健？是否遵守稀疏上下文策略？

评分模型

评分标准 
总分 = Routing (0-3) + Output (0-7)

建议阈值：
9-10：可直接复用
7-8：可用，观察稳定性
5-6：部分可用，需修正
<5  ：不通过，不可依赖
怎么用

📋 用 evals/cases/ 里的测试用例（7 个 case）给 AI 做测试

📝 用 evals/rubrics/ 里的 rubric 打分

🔄 每次接入新平台先跑 4 个基础 case + 1 个稀疏 case

⚠️ 不要因为 AI 会说漂亮话就给高分——看路由和判断是否正确
