# 署名案例 02：为什么我没有做一个提示词库，而是做了一套系统

## 背景

如果只是为了“让 AI 帮 PM 写东西”，最省事的做法其实很简单：

- 收集一批 Prompt
- 做几个模板
- 放几个常见场景

这样很快就能做出一个看起来内容很多的仓库。

但我没有停在这里。

## 我真正看到的问题

我观察到很多 AI for PM 的内容都有类似问题：

- 看起来什么都能做，但很难复用
- 输出很多，但判断不稳定
- 文案专业，但工作流不成体系
- 人能看，但 agent 不知道怎么正确调用

也就是说，问题不在于“有没有内容”，而在于：

**有没有一套能长期复用、能跨工具迁移、也能被 agent 正确执行的结构。**

## 我的核心判断

所以我做的主判断是：

**比起继续堆 Prompt，更值得做的是把 PM 工作方式沉淀成系统。**

这个系统至少要解决四件事：

1. 任务该怎么路由
2. 输出该怎么结构化
3. 工作流该怎么拆阶段
4. 不同平台接入后怎么评估稳定性

## 这就是为什么仓库后来长成现在这样

它不再只有 `skills/`，还增加了：

- `commands/`
- `agent/`
- `adapters/`
- `evals/`
- `private/`

每一层都对应一个真实问题：

### `skills/`

解决单点能力复用。

### `commands/`

解决真实 PM 工作不是单步任务，而是多阶段 workflow。

### `agent/`

解决“不同 agent 会不会乱用 skill、乱出结论”。

### `adapters/`

解决“这套系统能不能跨 Codex、Claude Code、Cursor 使用”。

### `evals/`

解决“回答看起来专业，不代表真的可靠”。

### `private/`

解决“怎么让一个通用系统慢慢长成更像你自己的系统”。

## 我为什么觉得这件事重要

因为 AI 时代最容易被快速复制的，是浅层产出。

最不容易被复制的，是：

- 问题怎么定义
- 判断怎么收敛
- 证据怎么组织
- 工作流怎么设计

如果产品经理只把 AI 当成一个更快写文档的工具，那最后最容易被替代的，也恰恰是这层浅层工作。

所以我更想做的是：

**把优秀 PM 的判断方式和工作流结构化，而不是只加速表面产出。**

## 这个案例的价值

这不是一个“我做了个仓库”的故事。

它真正体现的是一种方法论判断：

- 我选择系统，而不是内容堆积
- 我选择路由和评估，而不是只做输出
- 我选择长期可复用性，而不是一次性可展示性

这更像一个长期个人品牌，而不是一个短期内容项目。

## 一句话版本

**如果 AI 只让产品经理写得更快，它的价值很快会被抹平；如果 AI 能帮助产品经理把判断和工作流系统化，它才会成为真正的能力放大器。**
