# Messaging Hierarchy

This file keeps external language tight.

If we use too many “main ideas” at once, readers remember none of them.

## The Stack

### Category Term

`PM Operating System`

Use this when defining the category.

This is the phrase we want readers to mirror back.

### Product Name

`PM AI Skill Toolkit`

Use this when naming the repository, toolkit, and public product layer.

### Evaluation Layer

`PM Agent Benchmark`

Use this when talking about routing quality, output quality, and model comparisons.

### Workflow Terms

Use these as sub-terms, not top-level headlines:

- `Agent-native PM Workflow`
- `Commercialization Review Framework`
- `Pricing Packaging Brief`

## One-Line Rule

When writing outward-facing copy, use this shape:

`PM AI Skill Toolkit is a PM Operating System for product managers and AI agents.`

Do not invert it into:

- “I built a toolkit”
- “I built a prompt library”
- “I built a PM skills repo”

Those phrases shrink the project.

## Where Each Term Belongs

- README hero: product name + category term
- launch post first paragraph: category term first
- benchmark page: evaluation layer first
- scenario demos: workflow term first

## What To Avoid

- rotating the main phrase every post
- calling everything a toolkit
- using “prompt library” even as a lazy shorthand
- introducing new category terms before the old one sticks
